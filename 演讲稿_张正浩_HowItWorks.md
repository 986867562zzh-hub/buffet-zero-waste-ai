## Slide 6：AI 架构流程图（约 50 秒）

> 🖼️ PPT内容：一张从左到右的流程图
> 用户拍照 → AI引擎路由 → DeepSeek Vision → 感知哈希匹配 → PIL颜色分析
> 下方标注：识别结果（菜品名 + 热量 + 置信度）

**讲稿**：

"接下来我给大家讲一下我们这个系统最核心的部分——AI 是怎么工作的。

整个流程其实很简单：用户用手机拍一张餐盘照片，照片传到我们的服务器之后，会经过一个 **AI引擎路由器**。

它会按照优先级依次尝试：
- 第一优先是 **DeepSeek Vision API**——这是我们主力的 AI 视觉模型，性价比高，中文理解能力强；
- 如果 DeepSeek 不可用，会降级到 **感知哈希匹配**——这是我们自己写的一个算法，用图像的指纹特征跟菜品库做比对；
- 最后一层兜底是 **PIL 颜色分析**——保证在任何情况下都能返回一个合理的结果。

最终输出三样东西：**这道菜叫什么、大概多少热量、匹配置信度有多高**。

这个多层降级设计的好处是：不管有没有 API Key、不管网络好不好，系统都能跑。"

## Slide 7：AI 引擎详解（约 50 秒）

> 🖼️ PPT内容：三个引擎卡片并列
> DeepSeek Vision | 感知哈希 | PIL兜底
> 每个卡片下列出：技术原理、优点、缺点

**讲稿**：

"来，我们看每一层的细节。

**最上层 DeepSeek Vision**：用的是 DeepSeek 的多模态模型，我们把照片转成 base64 编码，连同一段精心设计的 prompt 一起发给 API。DeepSeek 会返回结构化的 JSON——菜名、分类、热量估算。优点是准确率非常高，中文菜名识别尤其好。

**中间层的感知哈希匹配**：这是我们自己实现的一个算法。原理是把图片缩放到 8×8 的灰度图，跟 12 道参考菜品逐一比对，算汉明距离。同时还会结合颜色直方图和边缘纹理特征，三个维度加权评分。这个不用花一分钱 API 费用，但准确率也还不错。

**最底层 PIL 颜色分析**：就是传统的图像处理——检测照片里红棕色的区域可能是肉类，绿色的区域可能是蔬菜。虽然粗糙，但至少不会瞎编一个菜名出来。

这三层加起来，保证系统在任何情况下都能给出诚实的、有依据的识别结果。"

> ⏱️ 翻页提示："那我具体展示一下这两个核心功能是怎么运作的。"

## Slide 8：功能 1+2 详解（约 50 秒）

> 🖼️ PPT内容：两个功能对比表格
> 热量追踪（餐前）| 剩食评分（餐后）
> input/output/处理逻辑

**讲稿**：

"我们的系统有两个使用场景，用同一个拍照入口切换。

**第一个是热量追踪**——用在餐前。你去自助餐台拿了一盘菜，拍照上传，AI 识别出这是什么菜、大概多少热量，然后这些热量会**跨多次拍照累积**起来。当累积热量达到你设定的阈值——默认是 1800 千卡——系统会弹出一个卡通角色提醒你：'饱饱啦，再拿可能吃不下哦~'。这其实是一个行为干预设计，用温柔的方式提醒食客不要拿太多。

**第二个是剩食评分**——用在餐后。吃完了拍一张餐盘，AI 识别盘子里剩了多少，生成一个 0-100 的浪费评分。95 分以上是'光盘英雄'，可以享受 85 折。这就是用经济激励来鼓励光盘行动。

两个功能一个管预防、一个管善后，合在一起就是一个完整的零废弃闭环。"

> ⏱️ 翻页提示："接下来是老师特别要求的——AI 证据展示。"

## Slide 9：Prompt 证据 + API 配置（约 50 秒）

> 🖼️ PPT内容：
> 左半：完整 DeepSeek Prompt（代码块样式）
> 右半：API 配置参数 + 返回JSON示例
> 底部：input 照片缩略图 → output JSON 截图

**讲稿**：

"这一页是我们 AI 工作的直接证据。

左边是我们发给 DeepSeek API 的**完整 prompt**。注意几个关键设计：
- 我们明确告诉 AI '你是一个专业的美食识别助手'——这是角色设定；
- 要求返回纯 JSON，不要 markdown 代码块——这保证了程序能直接解析；
- '只列出你实际看到的食物，诚实识别'——这个指令很重要，防止 AI 编造菜品。

右边是 API 的配置参数：temperature 设成 0.1，表示我们想要稳定一致的结果，不要创造性；max_tokens 2000，足够返回详细的分析。

底部是一个真实的输入输出对——左边是上传的锅包肉照片，右边是 API 返回的 JSON，包含了菜名、分类、热量估算和置信度。

这就是为什么我们敢说这个系统不是靠随机猜测——每一步都有 AI 在认真分析。"

> 🖼️ PPT内容：
> 左上：DeepSeekVision 类的代码截图（高亮关键行）
> 右上：SmartMatcher 感知哈希算法核心代码
> 底部：3组 input-output 示例（照片→识别结果）

**讲稿**：

"这一页展示一些关键代码，证明我们确实是自己写的。

左上角是 `DeepSeekVision` 类的核心代码——你们可以看到它用的是 OpenAI 兼容接口，base_url 指向 api.deepseek.com。这就是为什么我们能用 openai 这个 Python 库来调 DeepSeek 的模型。

右上角是感知哈希算法的核心实现——只有十几行代码，把图片缩到 8 乘 8，转灰度，然后跟平均值比较，生成一个 64 位的二进制指纹。两张图片的指纹越像，汉明距离越小，说明越可能是同一道菜。

底下是三个真实的输入输出示例。锅包肉的照片进去，AI 返回锅包肉 310 千卡。红烧肉进去，返回红烧肉 380 千卡。如果照片太模糊，系统会标一个低置信度并在结果里提示用户。

这些代码都在我们 GitHub 仓库里，链接在最后一页。"

## Slide 11：迭代记录 v1.0 → v2.5（约 50 秒）

> 🖼️ PPT内容：时间线形式
> v1.0(6/10): 随机生成 → v2.0(6/11): 固定菜品库+闭集匹配
> v2.1-v2.3: 热量追踪+感知哈希 → v2.4-v2.5(6/15): DeepSeek+演示模式+P1-P4

**讲稿**：

"最后给大家看一下我们的迭代过程，这个也体现了我们确实在不断测试和改进。

**v1.0 版本**是 6 月 10 号晚上做出来的——当时 AI 识别用的是随机生成结果。测试的时候我们发现，明明照片里没有的菜，它也能'识别'出来。这显然不行，我们用户测试的反馈就是'这不欺诈吗'。

所以 **v2.0** 彻底改了架构——建立了 12 道菜的固定菜品库，每道菜上传参考照片，AI 只在 12 道菜里面匹配，不会编造。

之后的 v2.1 到 v2.3，我们陆续加了热量累积追踪、用感知哈希替代纯颜色匹配、还有用 base64 把参考图存进 JSON 解决 Render 部署丢失的问题。

到 **v2.4 和 v2.5**，我们接入了 DeepSeek API 大幅提升识别准确率，做了一个一键演示模式方便课堂展示，还重构了剩菜搭配的推荐引擎——素食者不会收到肉的推荐，过敏信息会被真正过滤，3 个推荐套餐也做了差异化。

从 v1 到 v2.5，6 天时间，6 个大版本，踩了很多坑，但也收获很多。这就是我们真实的开发过程。"

## 🎯 演讲小贴士

1. **语速控制**：每个 Slide 控制在 50 秒左右，6 页刚好 5 分钟
2. **手势配合**：讲架构图时用手指流动方向；讲代码时指具体行
3. **眼神交流**：不要一直盯着屏幕，多看看老师和同学
4. **过渡自然**：每页结尾预留一句过渡语，翻页不尴尬
5. **Demo 衔接**：最后一句明确交给下一位演示的同学
